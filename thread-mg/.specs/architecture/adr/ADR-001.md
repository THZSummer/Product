# ADR-001: 简单计时器基准测试方案

**状态**: PROPOSED

**创建日期**: 2026-03-27

**关联 Feature**: MVP 基准测试框架 (mvp-test)

---

## 背景

在 MVP 基准测试框架开发中，需要选择基准测试的实现方案：

1. **MVP 时间约束**：需要在 1-2 周内交付可运行的基准测试框架
2. **结果可信度需求**：需要在实现复杂度和结果可信度之间取得平衡
3. **规范要求**：spec.md 明确说明"MVP 阶段不使用 JMH 微基准测试框架"
4. **技术评估**：考虑了三种方案
   - 方案 A：简单计时器 + 手动指标收集
   - 方案 B：集成 JMH 微基准测试框架
   - 方案 C：混合方案（简单计时器 + JFR 监控）

---

## 决策

**采用简单计时器（`System.nanoTime()`）+ 手动指标收集方案，不引入 JMH 或其他基准测试框架。**

### 技术要点

1. **计时方式**：使用 `System.nanoTime()` 进行纳秒级计时
2. **预热处理**：手动实现预热轮次（3-5 次）消除 JIT 编译影响
3. **指标计算**：手动计算吞吐量、平均延迟、百分位延迟
4. **数据结构**：使用 `ArrayList<Long>` 收集延迟样本，排序后计算百分位
5. **并发控制**：每个场景×策略组合顺序执行，避免并发干扰

### 核心实现

```java
// 计时示例
long startTime = System.nanoTime();
// ... 执行任务 ...
long endTime = System.nanoTime();
long durationNanos = endTime - startTime;

// 吞吐量计算
double throughput = taskCount / (durationNanos / 1_000_000_000.0);

// 百分位计算
Collections.sort(latencySamples);
int p95Index = (int) (latencySamples.size() * 0.95);
double p95Latency = latencySamples.get(p95Index) / 1_000_000.0; // 转为 ms
```

## 后果

### 正面影响

| 影响 | 说明 |
|------|------|
| ✅ **快速交付** | 实现简单，预计 3-4 天完成核心功能 |
| ✅ **代码透明** | 完全掌握代码逻辑，便于调试和优化 |
| ✅ **零外部依赖** | 仅使用 Java 标准库，降低构建复杂度 |
| ✅ **灵活可控** | 可自定义预热策略、统计方法、输出格式 |
| ✅ **易于扩展** | 后续 Phase 2 可无缝升级到 JMH 或混合方案 |

### 负面影响

| 影响 | 说明 | 缓解措施 |
|------|------|----------|
| ⚠️ **精度略低** | 缺少 JMH 的优化（如 dead code elimination 防护） | 任务设计保证足够计算量，添加结果验证 |
| ⚠️ **需要手动处理预热** | 需要自己实现预热逻辑 | 遵循 JMH 预热原则，3-5 次预热轮次 |
| ⚠️ **统计计算自己实现** | 百分位、标准差等需手动计算 | 编写单元测试验证统计计算正确性 |
| ⚠️ **结果可信度略低** | 非业界标准方法 | MVP 阶段可接受，Phase 2 评估引入 JMH |

### 技术约束

1. **Java 版本**：需要 Java 21+（虚拟线程需求，与 ADR-002 一致）
2. **任务设计**：单个任务执行时间应 > 1ms，避免计时误差
3. **GC 影响**：增加预热轮次，多次迭代取平均值
4. **结果验证**：必须添加结果正确性验证（如矩阵乘法结果校验）

## 后续演进

### Phase 2 评估点

在 Phase 2 开始前，评估是否引入 JMH：

1. **评估时机**：当需要更精确的微基准测试结果时
2. **评估标准**：
   - 当前方案结果波动 > 10%
   - 需要与业界标准基准对比
   - 团队有足够时间学习 JMH
3. **迁移路径**：
   - 保留现有接口（Scenario, ThreadStrategy）
   - 在 BenchmarkRunner 内部集成 JMH
   - 渐进式迁移，不影响现有场景和策略
