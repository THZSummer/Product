# 需求挖掘分析 - 数据开放平台

**会话 ID**: feature-session-001  
**分析创建时间**: 2026-04-03  
**状态**: 进行中

---

## 一、需求概述

### 1.1 核心定位
**数据开放平台**是 open-app 体系下的子平台，聚焦 XX 通讯平台的数据开放管理，将企业内部 XX 平台的数据开放给企业内部其它三方平台消费使用。

### 1.2 目标用户
| 角色 | 职责 |
|------|------|
| 数据 Owner | 注册数据、生产数据 |
| 开放平台管理员 | 审批数据注册信息 |
| 三方平台业务方 | 订阅数据、消费数据 |

---

## 二、核心痛点

### 2.1 现状描述
- AI 大行其道，企业内部 XX 通讯平台的 AI 能力较为薄弱
- 平台本身的能力多数局限在平台内部使用
- 企业内其它三方平台没有很好的把业务融进来，利用平台的资源能力开展他们自己的业务

### 2.2 核心问题
**能力封闭**：XX 平台的数据和能力无法被企业内部其他三方平台有效利用，形成数据孤岛和能力孤岛。

### 2.3 目标状态
**生态开放**：让三方平台能利用 XX 平台资源开展业务，形成企业内部的能力生态。

---

## 三、关键决策记录

### 3.1 数据范围
**决策**: 待定，暂无明确优先级与范围

**洞察**:
- 平台需要具备通用性和扩展性，能够适配不同类型的数据域
- 建设思路是先搭平台框架，数据域后续逐步接入

### 3.2 数据开放形式
**决策**: 多种形式均有可能（API 查询、事件订阅、批量导出、数据同步、数据服务）

**洞察**:
- 平台需要支持多种开放形式，不能局限于单一模式
- 可复用企业内部消息平台，降低建设成本
- 参考竞品：飞书、钉钉

### 3.3 核心流程
**决策**: 确认流程框架
```
数据 Owner 注册 → 管理员审批 → 数据上架 → 三方平台订阅 → 消费数据 → 审计日志
```

**详细决策**:
| 确认点 | 决策 |
|--------|------|
| 注册信息 | 参考竞品 |
| 审批触发 | 注册和更新都需要审批（取决于业务决策） |
| 消费凭证 | 三方平台需创建应用，获取 AK/SK；消费个人数据可能需用户授权 |
| 数据生产 | 主要是系统自动生产，不依赖人工推送 |

### 3.4 与 open-app 的关系
**决策**: 
- 业务归属：子平台，归于 open-app 体系
- 技术架构：技术上单独的服务
- 实现方式：业务上增强

**现状**:
- open-app 现有能力（应用管理、权限、OAuth 2.0）有代码但服务化程度低
- 数据开放平台是 open-app 的能力增强，不是从零开始

## 四、待深入问题

### 4.1 数据 Owner 视角（已完成）
- **谁是数据 Owner？** ✅ 业务模块负责人（如 IM 业务模块负责人拥有群、聊天等消息数据对象）
- **审批权限？** ✅ 平台审批只对技术负责（数据符合平台规范即可），业务上是否开放由线下沟通决议
- **数量关系？** ✅ 通常是一个，不必过于纠结
- **开放动机？** ✅ 价值驱动（对本业务模块的价值、对 XX 通讯平台的价值、对企业整体的价值、对用户的价值、对企业内三方平台的价值）
- **顾虑？** ✅ 担心开放带来的安全等负面影响

### 4.2 消费方视角（部分完成）
- **平台类型？** ✅ 大多数是企业内部自研平台系统
- **部署方式？** ✅ 完全独立部署，互相之间可能有集成
- **用户群体？** ✅ 各平台拥有巨大的企业内部用户群体，但各自偏向自己业务领域的用户（除 XX 通讯系统面向所有用户）
- **具体例子？** ✅ 客服系统、人事系统、投票选举系统等
- **想用数据做什么？** ⏳ 待业务调研
- **当前替代方案？** ⏳ 待业务调研

### 4.3 AI 场景（部分完成）
- **XX 平台 AI 能力？** ✅ 内部有，但比较薄弱
- **核心问题？** ✅ 没有标准统一的通道获取 XX 平台数据
- **现状？** ✅ 可能已经在消费，只是没有标准统一的通道（所以需要数据开放平台做这个事情）
- **具体 AI 应用规划？** ⏳ 待调研

### 4.4 数据分类与敏感度（部分完成）
- **敏感度分级？** ✅ 逻辑或事实上肯定有
- **统一清单？** ❌ XX 平台内部没有统一的地方去定义或记录这些定义
- **数据对象清单？** ❌ XX 平台内部有哪些数据对象，也没有一个统一的清单
- **开放策略差异？** ✅ 不同级别自然存在开放策略的差异
- **明确不能开放的数据？** ⏳ 会随时间变化，不是静态的

**关键洞察**:
- 数据开放平台可能需要承担**数据资产目录**的职责
- 数据分级标准需要**可配置、可演进**，不是硬编码
- 这是一个**治理问题**，不只是技术问题

### 4.5 组织与流程（部分完成）
- **决策角色？** ✅ 数据 Owner、开放平台管理员、安全、技术、CTO 等
- **线上 vs 线下？** ✅ 安全、技术、CTO 等审批大概率作为线下业务约束，不沉淀到系统功能
- **流程参考？** ✅ 参考钉钉、飞书，估计类似
- **运营角色？** ✅ 应该有专门的平台运营人员

**关键洞察**:
- 线下流程 + 线上平台结合
- 平台聚焦技术审批，业务决策在线下
- 需要专门的平台运营团队

### 4.6 成功标准（部分完成）
- **定性指标？** ✅ 认可（数据 Owner 更方便、三方平台更容易、平台方更安全可控）
- **定量指标？** ✅ 系统会具备度量能力，但具体目标值取决于业务运营推广的投入力度
- **核心观点？** ✅ 前提将系统构建好，具备这些能力（系统是基础设施，业务指标取决于运营）

**关键洞察**:
- 系统提供能力，业务指标取决于运营
- 务实的建设思路：先有能力，再有规模

### 4.7 竞品对标细节（部分完成）
- **API 中心现状？** ✅ 已经有了（应用管理里有 API 订阅管理）
- **现有 API 特点？** ✅ 少量、偏业务功能（如发卡片）、API 文档集中在 open-app 文档中心
- **缺失能力？** ❌ 没有专注于数据开放方面的 API
- **优化方向？** ✅ 单人可参考飞书、钉钉的做法做优化
- **计费模式？** ❌ 可能不涉及计费（企业内部使用）
- **可参考维度？** ✅ 数据目录、数据注册流程、审批机制、权限粒度、开放形式、开发者体验等

**关键洞察**:
- 不是从零开始，是在现有 API 中心基础上增强
- 现有能力偏业务功能，需要增加数据开放能力
- 飞书/钉钉在数据开放维度的做法都可参考

## 七、新需求维度：数据价值赋能（2026-04-03 补充挖掘）

### 7.1 核心洞察

| 维度 | 关键发现 |
|------|---------|
| **使用方式** | 多样化：直接展示、数据同步、业务融合、流程集成、AI 应用、数据分析等 |

### 7.2 赋能需求优先级

**TOP 5 赋能支持**（按优先级排序）：
1. **数据字典/数据地图**：数据含义、关系、使用建议

3. **技术咨询支持**：快速响应、专业解答
4. **典型场景解决方案**：最佳实践、案例分享
5. **数据价值评估工具**：帮助消费方量化数据价值

**场景引导方式**（按优先级排序）：
1. **场景分类目录**：按业务场景分类展示数据（如：HR 场景、客服场景、AI 场景）
2. **案例库**：收集并分享成功案例
3. **解决方案模板**：提供标准化的解决方案模板

**数据理解支持**：
- 数据字典：每个字段的含义、类型、示例值
- 数据血缘：数据来源、加工逻辑、更新频率
- 使用建议：推荐使用场景、不推荐使用场景
- 数据预览：在线查看样本数据

**持续优化支持**：

## 八、阶段三需求挖掘：权限关系（2026-04-09）

### 8.1 核心设计原则

**双重权限模型**：
```
消费权限 = 通道权限 ⊗ 数据对象权限
```

| 维度 | 设计决策 |
|------|---------|
| **数据对象定义** | 业务逻辑层面概念（如：员工信息对象、文件对象、群组对象），不是数据库表级别 |
| **通道类型** | P0: REST API、Webhook、消息队列 |
| **权限主体** | 开放平台的应用（Application），永久有效，主动回收 |
| **审批流程** | 一次申请=一次审批，动态生成审批链 |
| **审批角色** | 数据 Owner、通道 Owner、平台管理员、上级/CTO（L3+）、安全合规（L3+） |
| **权限回收** | 低频谨慎，提前通知，平滑过渡 |
| **审计日志** | 以通道为入口记录，多用途支持 |
| **应用隔离** | 无共享/继承，岗位稳定，凭证无感 |

### 8.2 关键洞察

**1. 数据对象是业务概念，不是技术概念**
- 数据对象是业务逻辑层面的输出单元
- 示例：员工信息对象、文件对象、群组对象
- 不对应数据库实体，简化了权限管理复杂度

**2. 双重权限模型解决通道与数据的解耦**
- 通道权限和数据对象权限独立管理
- 应用需同时拥有两种权限才能消费
- 支持同一数据对象不同通道权限独立控制

**3. 应用级授权简化权限管理**
- 权限绑定在应用 ID 上，不是 AK/SK
- AK/SK 重置不影响权限
- 数据 Owner 人员变更不影响已授权权限（针对岗位）

**4. 动态审批链平衡效率与安全**
- 根据数据敏感度自动生成审批链
- L1-L2：2-3 级审批（数据 Owner+ 通道 Owner+ 平台管理员）
- L3-L4：4-5 级审批（增加上级/CTO+ 安全合规）
- 一次申请完成所有审批，用户体验友好

**5. 权限回收低频谨慎**
- 回收是严肃的决策，不轻易执行
- 必须提前通知应用方，给缓冲时间
- 逻辑数据对象层面变动频率低

**6. 审计以通道为入口**
- 不单独记录"权限使用日志"
- API 调用日志、事件推送日志关联权限信息
- 支持安全审计、使用量统计、性能优化、问题排查、合规要求

**7. 模块协同复用优先**
- 应用管理：复用现有能力
- API 开放：复用现有 API 订阅管理
- OAuth2：暂不涉及，仅管理 AK/SK
- 审计日志：复用统一审计能力

### 8.3 与前期决策的关系

| 前期决策 | 阶段三深化 |
|---------|-----------|
| 应用管理（AK/SK） | 明确权限基于应用 ID，AK/SK 仅是凭证 |
| 数据敏感度分级 | 明确审批链根据敏感度动态生成 |
| 全流程线上审批 | 明确一次申请=一次审批，动态生成审批链 |
| 审计日志 | 明确以通道为入口记录，多用途支持 |
| 复用现有能力 | 明确应用管理、API 开放、审计日志复用策略 |

### 8.4 待确认事项

| 事项 | 说明 |
|------|------|
| 可申请目录过滤 | 是否需要根据应用资质过滤可申请的权限 |
| 回收提前通知天数 | 提前 7 天、15 天还是 30 天通知 |
| 变更程度判定 | 什么程度的变更需要重新审批 |
