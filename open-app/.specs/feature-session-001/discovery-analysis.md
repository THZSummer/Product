# 需求挖掘分析 - 数据开放平台

**会话 ID**: feature-session-001  
**分析创建时间**: 2026-04-03  
**状态**: 进行中

---

## 一、需求概述

### 1.1 核心定位
**数据开放平台**是 open-app 体系下的子平台，聚焦 XX 通讯平台的数据开放管理，将企业内部 XX 平台的数据开放给企业内部其它三方平台消费使用。

### 1.2 目标用户
| 角色 | 职责 |
|------|------|
| 数据 Owner | 注册数据、生产数据 |
| 开放平台管理员 | 审批数据注册信息 |
| 三方平台业务方 | 订阅数据、消费数据 |

---

## 二、核心痛点

### 2.1 现状描述
- AI 大行其道，企业内部 XX 通讯平台的 AI 能力较为薄弱
- 平台本身的能力多数局限在平台内部使用
- 企业内其它三方平台没有很好的把业务融进来，利用平台的资源能力开展他们自己的业务

### 2.2 核心问题
**能力封闭**：XX 平台的数据和能力无法被企业内部其他三方平台有效利用，形成数据孤岛和能力孤岛。

### 2.3 目标状态
**生态开放**：让三方平台能利用 XX 平台资源开展业务，形成企业内部的能力生态。

---

## 三、关键决策记录

### 3.1 数据范围
**决策**: 待定，暂无明确优先级与范围

**洞察**:
- 平台需要具备通用性和扩展性，能够适配不同类型的数据域
- 建设思路是先搭平台框架，数据域后续逐步接入

### 3.2 数据开放形式
**决策**: 多种形式均有可能（API 查询、事件订阅、批量导出、数据同步、数据服务）

**洞察**:
- 平台需要支持多种开放形式，不能局限于单一模式
- 可复用企业内部消息平台，降低建设成本
- 参考竞品：飞书、钉钉

### 3.3 核心流程
**决策**: 确认流程框架
```
数据 Owner 注册 → 管理员审批 → 数据上架 → 三方平台订阅 → 消费数据 → 审计日志
```

**详细决策**:
| 确认点 | 决策 |
|--------|------|
| 注册信息 | 参考竞品 |
| 审批触发 | 注册和更新都需要审批（取决于业务决策） |
| 消费凭证 | 三方平台需创建应用，获取 AK/SK；消费个人数据可能需用户授权 |
| 数据生产 | 主要是系统自动生产，不依赖人工推送 |

### 3.4 与 open-app 的关系
**决策**: 
- 业务归属：子平台，归于 open-app 体系
- 技术架构：技术上单独的服务
- 实现方式：业务上增强

**现状**:
- open-app 现有能力（应用管理、权限、OAuth 2.0）有代码但服务化程度低
- 数据开放平台是 open-app 的能力增强，不是从零开始

## 四、待深入问题

### 4.1 数据 Owner 视角（已完成）
- **谁是数据 Owner？** ✅ 业务模块负责人（如 IM 业务模块负责人拥有群、聊天等消息数据对象）
- **审批权限？** ✅ 平台审批只对技术负责（数据符合平台规范即可），业务上是否开放由线下沟通决议
- **数量关系？** ✅ 通常是一个，不必过于纠结
- **开放动机？** ✅ 价值驱动（对本业务模块的价值、对 XX 通讯平台的价值、对企业整体的价值、对用户的价值、对企业内三方平台的价值）
- **顾虑？** ✅ 担心开放带来的安全等负面影响

### 4.2 消费方视角（部分完成）
- **平台类型？** ✅ 大多数是企业内部自研平台系统
- **部署方式？** ✅ 完全独立部署，互相之间可能有集成
- **用户群体？** ✅ 各平台拥有巨大的企业内部用户群体，但各自偏向自己业务领域的用户（除 XX 通讯系统面向所有用户）
- **具体例子？** ✅ 客服系统、人事系统、投票选举系统等
- **想用数据做什么？** ⏳ 待业务调研
- **当前替代方案？** ⏳ 待业务调研

### 4.3 AI 场景（部分完成）
- **XX 平台 AI 能力？** ✅ 内部有，但比较薄弱
- **核心问题？** ✅ 没有标准统一的通道获取 XX 平台数据
- **现状？** ✅ 可能已经在消费，只是没有标准统一的通道（所以需要数据开放平台做这个事情）
- **具体 AI 应用规划？** ⏳ 待调研

### 4.4 数据分类与敏感度（部分完成）
- **敏感度分级？** ✅ 逻辑或事实上肯定有
- **统一清单？** ❌ XX 平台内部没有统一的地方去定义或记录这些定义
- **数据对象清单？** ❌ XX 平台内部有哪些数据对象，也没有一个统一的清单
- **开放策略差异？** ✅ 不同级别自然存在开放策略的差异
- **明确不能开放的数据？** ⏳ 会随时间变化，不是静态的

**关键洞察**:
- 数据开放平台可能需要承担**数据资产目录**的职责
- 数据分级标准需要**可配置、可演进**，不是硬编码
- 这是一个**治理问题**，不只是技术问题

### 4.5 组织与流程（部分完成）
- **决策角色？** ✅ 数据 Owner、开放平台管理员、安全、技术、CTO 等
- **线上 vs 线下？** ✅ 安全、技术、CTO 等审批大概率作为线下业务约束，不沉淀到系统功能
- **流程参考？** ✅ 参考钉钉、飞书，估计类似
- **运营角色？** ✅ 应该有专门的平台运营人员

**关键洞察**:
- 线下流程 + 线上平台结合
- 平台聚焦技术审批，业务决策在线下
- 需要专门的平台运营团队

### 4.6 成功标准（部分完成）
- **定性指标？** ✅ 认可（数据 Owner 更方便、三方平台更容易、平台方更安全可控）
- **定量指标？** ✅ 系统会具备度量能力，但具体目标值取决于业务运营推广的投入力度
- **核心观点？** ✅ 前提将系统构建好，具备这些能力（系统是基础设施，业务指标取决于运营）

**关键洞察**:
- 系统提供能力，业务指标取决于运营
- 务实的建设思路：先有能力，再有规模

### 4.7 竞品对标细节（部分完成）
- **API 中心现状？** ✅ 已经有了（应用管理里有 API 订阅管理）
- **现有 API 特点？** ✅ 少量、偏业务功能（如发卡片）、API 文档集中在 open-app 文档中心
- **缺失能力？** ❌ 没有专注于数据开放方面的 API
- **优化方向？** ✅ 单人可参考飞书、钉钉的做法做优化
- **计费模式？** ❌ 可能不涉及计费（企业内部使用）
- **可参考维度？** ✅ 数据目录、数据注册流程、审批机制、权限粒度、开放形式、开发者体验等

**关键洞察**:
- 不是从零开始，是在现有 API 中心基础上增强
- 现有能力偏业务功能，需要增加数据开放能力
- 飞书/钉钉在数据开放维度的做法都可参考

## 五、风险与假设

### 5.1 关键假设
- 假设数据 Owner 有开放数据的意愿
- 假设企业内部三方平台有使用 XX 平台数据的需求
- 假设现有 open-app 代码可以服务化改造

### 5.2 潜在风险
- 数据 Owner 可能担心数据安全和责任问题
- 数据敏感度分级可能影响开放策略
- 服务化改造的工作量可能超出预期
