# ADR-001: RBAC 权限模型设计

## 状态
PROPOSED

## 背景
open-app 需要实现访问控制机制。需要在多种权限模型中做出选择。

## 选项对比

### 选项 1: ACL (Access Control List)
**优点:**
- 简单直观
- 适合小规模系统
- 每个资源独立控制

**缺点:**
- 用户增多时管理复杂
- 权限变更需要修改多个 ACL
- 不易审计

### 选项 2: RBAC (Role-Based Access Control)
**优点:**
- 通过角色间接管理权限
- 用户增减不影响权限配置
- 易于审计和合规
- 行业标准，文档丰富

**缺点:**
- 角色爆炸问题（角色过多）
- 不支持细粒度数据权限
- 角色层级需要额外设计

### 选项 3: ABAC (Attribute-Based Access Control)
**优点:**
- 最灵活，支持复杂规则
- 支持数据级权限
- 动态决策

**缺点:**
- 实现复杂
- 性能开销大
- 规则管理困难

## 决策
选择 **选项 2: RBAC** 作为核心模型，预留 ABAC 扩展点。

### 理由
1. 符合大多数企业权限管理需求
2. 实现复杂度适中
3. 团队熟悉 RBAC 概念
4. 可以通过"数据范围"扩展支持部分 ABAC 场景

## 实施细节

### 核心表结构
```
users ─┬─ user_roles ── roles ─┬─ role_permissions ── permissions
       │                        │
       └─ (直接权限，可选)       └─ (内置角色标记)
```

### 权限代码规范
```
格式：{module}:{action}
示例：
- user:create
- user:read
- role:delete
- report:export
```

### 内置角色
| 角色 | 说明 | 可删除 |
|------|------|--------|
| super_admin | 超级管理员，拥有所有权限 | 否 |
| admin | 系统管理员 | 否 |
| viewer | 只读用户 | 否 |

## 后果

### 正面
- 权限管理清晰
- 用户 - 角色解耦
- 易于审计

### 负面
- 需要处理角色爆炸问题
- 数据级权限需要额外设计

### 风险缓解
1. 制定角色命名规范，避免随意创建
2. 预留 `data_scope` 字段用于未来数据权限
3. 定期审计角色使用情况

## 扩展计划

### v1.0 - 基础 RBAC
- 用户 - 角色 - 权限三层模型
- 基础 CRUD

### v1.1 - 角色增强
- 角色层级（继承）
- 角色有效期

### v2.0 - ABAC 混合
- 数据范围（部门、项目）
- 动态规则引擎
