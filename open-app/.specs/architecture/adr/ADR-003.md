# ADR-003: 前端状态管理方案选择

## 状态

PROPOSED

## 背景

应用管理 Web 端需要在前端进行状态管理，以协调组件间的数据流和 API 调用。需要在多种状态管理方案中做出选择。

本模块的状态管理需求：
- 应用列表数据（分页、筛选、排序）
- 单个应用详情数据
- UI 状态（loading、error、modal 可见性）
- 表单数据（创建/编辑）

## 选项对比

### 选项 1: Zustand

**描述**: 轻量级状态管理库，基于 Hook 模式

**优点:**
- 极简 API，学习成本低（~1KB）
- 无需 Provider 包裹，直接使用 Hook
- TypeScript 支持优秀，类型推导完整
- 支持中间件（persist、devtools、immer）
- 无样板代码，开发效率高
- 支持选择器优化重渲染

**缺点:**
- 生态不如 Redux 丰富
- 大型应用最佳实践较少
- 社区资源相对有限
- 持久化需要额外配置

**适用场景:**
- 中小型应用
- 追求简洁和开发效率
- TypeScript 项目

**工作量评估:** 2 天

---

### 选项 2: React Context + useReducer

**描述**: React 原生状态管理方案

**优点:**
- 无需额外依赖
- React 官方推荐方案
- 易于理解和调试
- 与 React 生态完全兼容

**缺点:**
- 性能优化复杂（需要 useMemo/useCallback）
- 代码冗余较多（Action、Reducer、Context）
- 跨组件共享状态不便
- 不支持 DevTools 开箱即用
- 全局状态需要多层 Provider 包裹

**适用场景:**
- 小型应用
- 不希望引入额外依赖
- 简单状态管理需求

**工作量评估:** 3 天

---

### 选项 3: Redux Toolkit

**描述**: 标准 Redux 的现代化版本

**优点:**
- 生态最丰富，插件众多
- DevTools 支持最好
- 大型应用最佳实践多
- 中间件生态完善（redux-thunk、redux-saga）
- 团队熟悉度高

**缺点:**
- 代码冗余（尽管有 Toolkit 简化）
- 学习曲线较陡
- Bundle 大小较大（~20KB + 依赖）
- 样板代码仍然较多
- 对于本模块规模过度设计

**适用场景:**
- 大型复杂应用
- 需要复杂中间件
- 团队已有 Redux 经验

**工作量评估:** 4 天

---

### 选项 4: Jotai

**描述**: 原子化状态管理库

**优点:**
- 原子化设计，粒度细
- 性能优秀（细粒度更新）
- TypeScript 支持好
- API 简洁

**缺点:**
- 原子过多时管理复杂
- 学习曲线较陡（原子概念）
- 社区相对较小
- 不适合列表/集合类状态

**适用场景:**
- 需要细粒度状态更新
- 复杂交互场景
- 性能敏感应用

**工作量评估:** 3 天

---

## 决策

选择 **选项 1: Zustand** 作为前端状态管理方案。

### 理由

1. **符合模块规模**: 本模块为中等规模前端应用（4 页面 + 10 组件），Zustand 的轻量特性正好匹配
2. **性能指标**: Bundle 大小仅 ~1KB，符合 Spec 要求（< 500KB gzip）
3. **开发效率**: API 简洁，无样板代码，可快速开发
4. **TypeScript 友好**: 类型推导完整，符合 Spec 的 TypeScript 要求
5. **团队因素**: 学习成本低，易于上手
6. **扩展性**: 支持中间件，v2.0 可添加 persist 等高级功能

### 不选其他方案的理由

- **不选 Context**: 性能优化复杂，长期维护成本高
- **不选 Redux**: 对于本模块规模过度设计，Bundle 过大
- **不选 Jotai**: 原子化设计不适合列表/集合类状态管理

## 实施细节

### Store 结构设计

```typescript
// stores/applicationStore.ts
import { create } from 'zustand';

interface ApplicationState {
  // 列表数据
  applications: Application[];
  total: number;
  page: number;
  pageSize: number;
  
  // 筛选条件
  filters: {
    keyword?: string;
    status?: AppStatus;
    type?: AppType;
  };
  
  // 当前选中的应用
  currentApplication?: Application;
  
  // UI 状态
  loading: boolean;
  error?: string;
  
  // 操作
  fetchApplications: (params: ListParams) => Promise<void>;
  createApplication: (data: CreateAppData) => Promise<Application>;
  updateApplication: (id: string, data: UpdateAppData) => Promise<Application>;
  deleteApplication: (id: string) => Promise<void>;
  setCurrentApplication: (app?: Application) => void;
  setFilters: (filters: Partial<ApplicationState['filters']>) => void;
  reset: () => void;
}

export const useApplicationStore = create<ApplicationState>((set, get) => ({
  // 初始状态
  applications: [],
  total: 0,
  page: 1,
  pageSize: 20,
  filters: {},
  loading: false,
  
  // 列表获取
  fetchApplications: async (params) => {
    set({ loading: true, error: undefined });
    try {
      const response = await applicationApi.getList(params);
      set({ 
        applications: response.data, 
        total: response.total,
        page: params.page,
        pageSize: params.pageSize,
        loading: false 
      });
    } catch (error) {
      set({ error: error.message, loading: false });
      throw error;
    }
  },
  
  // 创建应用
  createApplication: async (data) => {
    set({ loading: true, error: undefined });
    try {
      const application = await applicationApi.create(data);
      set({ loading: false });
      // 刷新列表
      get().fetchApplications({ page: 1, pageSize: get().pageSize });
      return application;
    } catch (error) {
      set({ error: error.message, loading: false });
      throw error;
    }
  },
  
  // 更新应用
  updateApplication: async (id, data) => {
    set({ loading: true, error: undefined });
    try {
      const application = await applicationApi.update(id, data);
      set({ currentApplication: application, loading: false });
      return application;
    } catch (error) {
      set({ error: error.message, loading: false });
      throw error;
    }
  },
  
  // 删除应用
  deleteApplication: async (id) => {
    set({ loading: true, error: undefined });
    try {
      await applicationApi.delete(id);
      set({ loading: false });
      // 刷新列表
      get().fetchApplications({ page: 1, pageSize: get().pageSize });
    } catch (error) {
      set({ error: error.message, loading: false });
      throw error;
    }
  },
  
  // 设置当前应用
  setCurrentApplication: (app) => {
    set({ currentApplication: app });
  },
  
  // 设置筛选条件
  setFilters: (filters) => {
    set((state) => ({ 
      filters: { ...state.filters, ...filters },
      page: 1 // 重置页码
    }));
  },
  
  // 重置
  reset: () => {
    set({
      applications: [],
      total: 0,
      page: 1,
      pageSize: 20,
      filters: {},
      currentApplication: undefined,
      loading: false,
      error: undefined,
    });
  },
}));
```

### 组件使用示例

```typescript
// pages/ApplicationList.tsx
import { useApplicationStore } from '@/stores/applicationStore';

export const ApplicationList = () => {
  const { 
    applications, 
    total, 
    loading, 
    fetchApplications,
    deleteApplication 
  } = useApplicationStore();
  
  // 使用选择器优化重渲染
  const apps = useApplicationStore((state) => state.applications);
  
  return (
    <div>
      {loading && <Spinner />}
      <AppTable 
        data={apps}
        onDelete={deleteApplication}
      />
      <Pagination total={total} />
    </div>
  );
};
```

### 持久化配置（v2.0 扩展）

```typescript
import { create } from 'zustand';
import { persist, createJSONStorage } from 'zustand/middleware';

export const useApplicationStore = create(
  persist(
    (set, get) => ({
      // ... state and actions
      filters: {}, // 持久化筛选条件
    }),
    {
      name: 'app-storage',
      storage: createJSONStorage(() => sessionStorage),
      partialize: (state) => ({ filters: state.filters }), // 只持久化部分状态
    }
  )
);
```

## 后果

### 正面

- **开发效率提升**: 无样板代码，API 简洁
- **Bundle 大小优化**: Zustand 仅 ~1KB
- **TypeScript 体验好**: 完整的类型推导
- **易于测试**: 纯函数式 API，易于 Mock

### 负面

- **生态限制**: 中间件和插件不如 Redux 丰富
- **团队学习成本**: 需要学习新工具（但学习曲线低）
- **长期维护**: 大型应用最佳实践较少

### 风险缓解

1. **团队培训**: 提供 Zustand 使用文档和示例
2. **代码规范**: 制定 Store 编写规范，确保一致性
3. **性能监控**: 监控重渲染次数，必要时优化
4. **扩展计划**: v2.0 评估是否需要升级到更复杂方案
