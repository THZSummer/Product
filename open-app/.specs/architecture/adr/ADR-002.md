# ADR-002: 应用管理模块缓存架构设计

## 状态

PROPOSED

## 背景

应用管理模块需要支持高并发查询场景（P95 < 100ms），同时减轻数据库压力。需要设计合理的缓存策略。

## 选项对比

### 选项 1: 无缓存

**优点:**
- 实现简单
- 数据一致性最好
- 无需维护缓存基础设施

**缺点:**
- 数据库压力大
- 查询延迟无法保证
- 无法应对突发流量

### 选项 2: 应用层缓存（Redis）

**优点:**
- 查询性能高（P95 < 10ms）
- 减轻数据库压力
- 支持分布式部署
- 灵活的过期策略

**缺点:**
- 增加系统复杂度
- 缓存一致性问题
- 需要额外的运维成本
- 缓存穿透/雪崩/击穿风险

### 选项 3: 数据库层缓存（PostgreSQL Buffer）

**优点:**
- 无需额外组件
- 数据一致性好
- 配置简单

**缺点:**
- 性能提升有限
- 无法应对高并发
- 应用重启后缓存丢失

### 选项 4: 多级缓存（本地 + Redis）

**优点:**
- 最高性能
- Redis 故障时可降级
- 减少 Redis 网络开销

**缺点:**
- 实现复杂
- 本地缓存一致性问题
- 多实例缓存冗余

## 决策

选择 **选项 2: 应用层缓存（Redis）** 作为 v1.0 缓存方案。

### 理由

1. **性能与复杂度平衡**：Redis 缓存能满足性能需求（P95 < 100ms），同时复杂度可控
2. **团队熟悉度**：团队已有 Redis 使用经验
3. **基础设施支持**：项目已规划 Redis 基础设施
4. **可扩展性**：v2.0 可升级为多级缓存

### 不选多级缓存的理由

v1.0 业务规模较小，多级缓存的收益不明显，但会显著增加复杂度。

## 实施细节

### 缓存 Key 设计规范

```
格式：app:{type}:{identifier}

示例:
- app:detail:app_xxxxxxxxxxxx      # 应用详情
- app:list:user_xxxxxxxxxxxx:1     # 应用列表（page=1）
- app:name:user_xxxxxxxxxxxx:myapp # 名称唯一性检查
```

### 缓存数据结构

```typescript
// 应用详情
{
  "key": "app:detail:app_xxx",
  "value": {
    "id": "app_xxx",
    "name": "应用名称",
    "status": "active",
    // ... 其他字段
  },
  "ttl": 300
}

// 应用列表（带分页和筛选）
{
  "key": "app:list:user_xxx:active:self_build:1:20",
  "value": {
    "data": [...],
    "total": 100,
    "page": 1,
    "pageSize": 20
  },
  "ttl": 60
}
```

### 缓存失效策略

| 操作 | 失效的缓存 |
|------|-----------|
| 创建应用 | 用户的应用列表缓存 |
| 更新应用 | 应用详情缓存、用户列表缓存 |
| 删除应用 | 应用详情缓存、用户列表缓存 |
| 状态变更 | 应用详情缓存、相关列表缓存 |
| 名称变更 | 名称唯一性检查缓存 |

### 缓存穿透防护

```typescript
// 空值缓存（短时间）
async function getApplication(id: string): Promise<Application | null> {
  const cacheKey = `app:detail:${id}`;
  
  // 尝试从缓存获取
  const cached = await redis.get(cacheKey);
  if (cached !== null) {
    return cached === 'NULL' ? null : JSON.parse(cached);
  }
  
  // 从数据库获取
  const app = await repository.findById(id);
  
  // 空值也缓存，避免穿透
  if (app === null) {
    await redis.setex(cacheKey, 60, 'NULL'); // 空值缓存 1 分钟
    return null;
  }
  
  // 正常缓存
  await redis.setex(cacheKey, 300, JSON.stringify(app));
  return app;
}
```

### 缓存雪崩防护

```typescript
// TTL 添加随机偏移量
function getTTL(baseTTL: number): number {
  const jitter = Math.floor(Math.random() * 60); // 0-60 秒随机
  return baseTTL + jitter;
}

// 使用示例
await redis.setex(cacheKey, getTTL(300), JSON.stringify(app));
```

## 后果

### 正面

- 查询性能提升至 P95 < 50ms
- 数据库负载降低 60-80%
- 支持更高的并发量

### 负面

- 增加 Redis 依赖
- 需要考虑缓存一致性问题
- 增加运维复杂度

### 风险缓解

1. **缓存一致性**：采用失效策略而非更新策略，确保数据一致性
2. **Redis 故障**：降级到数据库查询（性能下降但可用）
3. **缓存穿透**：空值缓存 + 布隆过滤器（如需要）
4. **监控告警**：监控缓存命中率，低于阈值时告警

## 监控指标

| 指标 | 阈值 | 告警级别 |
|------|------|----------|
| 缓存命中率 | < 70% | Warning |
| 缓存命中率 | < 50% | Critical |
| Redis 内存使用率 | > 80% | Warning |
| Redis 内存使用率 | > 90% | Critical |
| 缓存平均响应时间 | > 10ms | Warning |
