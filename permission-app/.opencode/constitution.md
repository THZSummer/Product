# SDD Constitution - 项目宪法

这是本项目的不可协商原则。所有代码和规范必须符合这些规则。

## 架构原则

- [ ] 遵循单一职责原则
- [ ] 模块间依赖必须单向
- [ ] 禁止循环依赖
- [ ] 公共 API 必须稳定且有版本控制
- [ ] 权限检查必须集中管理

## 代码规范

- [ ] 所有公共函数必须有类型注解
- [ ] 禁止使用 `any` 类型（除非有明确理由并注释）
- [ ] 函数长度不超过 50 行
- [ ] 文件长度不超过 500 行
- [ ] 权限相关代码必须有代码审查

## 测试要求

- [ ] 所有权限逻辑必须有单元测试
- [ ] 核心业务逻辑测试覆盖率 ≥ 80%
- [ ] 权限边界必须有集成测试
- [ ] 测试必须独立可运行

## 安全标准

- [ ] 所有用户输入必须验证
- [ ] 权限检查必须在服务端执行（不能仅依赖前端）
- [ ] 敏感操作必须有审计日志
- [ ] 禁止硬编码密钥或权限
- [ ] 权限变更必须有审批流程

## 权限设计原则

- [ ] 遵循最小权限原则（Principle of Least Privilege）
- [ ] 权限必须可组合（RBAC/ABAC）
- [ ] 权限变更必须可追溯
- [ ] 默认拒绝，显式允许

## 文档要求

- [ ] 公共 API 必须有文档注释
- [ ] 权限模型必须有架构文档
- [ ] 规范必须先行于实现
- [ ] 变更必须有更新记录

## 依赖管理

- [ ] 新增依赖需要评估和记录
- [ ] 优先使用项目已有依赖
- [ ] 禁止使用不再维护的库
- [ ] 定期审计依赖安全漏洞
